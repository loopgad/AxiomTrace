# AxiomTrace 有线格式规范

> 版本：v1.0  |  状态：**已冻结**  |  对应代码：`baremetal/core/axiom_event.c`, `baremetal/core/axiom_crc.c`

---

## 1. 范围

本文定义了 AxiomTrace 事件记录如何序列化为字节流，以便通过 UART、USB、CAN-FD 或内存缓冲区进行传输。

**自 v0.2-wire 版本起冻结**。任何修改都必须同步更新规范、Golden Frame、解码器，并通过完整的回归测试。

---

## 2. 帧结构

有线传输帧由以下部分组成：

```
[ 帧主体 ]
  ├── 报头 (8 字节)
  ├── 时间戳 (1..5 字节, 差分压缩)
  ├── 有效载荷长度 (1 字节)
  ├── 有效载荷 (0..255 字节)
  └── CRC-16 (2 字节)
[ 可选传输封装 ]
```

### 2.1 时间戳

时间戳字段由 `axiom_write()` 为每个事件自动插入。它使用相对于前一个事件原始时间戳（来自 `axiom_port_timestamp()`）的差分压缩：

| 差分范围 | 字节数 | 编码方式 |
|---------------|-------|----------|
| 0 – 127 | 1 | `0b0xxxxxxx` (7 位值) |
| 128 – 16,383 | 2 | `0b10xxxxxx` + 1 字节 (13 位值) |
| 16,384 – 2,097,151 | 3 | `0b110xxxxx` + 2 字节 (19 位值) |
| 2,097,152 – 4,294,967,295 | 5 | `0xFF` + 4 字节 (完整 32 位) |

时间戳包含在 CRC-16 计算范围内（报头 + 时间戳 + 有效载荷长度 + 有效载荷）。解码器必须先解码变长的时间戳字段，才能确定 `payload_len` 在 `frame[8 + ts_len]` 的位置。

### 2.2 字节序

除非传输层另有强制要求（例如 CAN-FD 惯例使用大端序；CAN-FD 后端会执行字节交换），否则所有多字节字段均采用 **小端序 (Little-endian)**。

### 2.3 对齐

有线格式是非对齐的（紧凑型）。编码器使用 `memcpy` 或按字节写入，以避免在不支持非对齐访问的架构上出现未定义行为。解码器也必须遵循此原则。

---

## 3. COBS 编码（字节流传输）

对于 UART 和 USB CDC，整个“帧主体”会进行 **COBS 编码**，以消除帧内部的 `0x00` 字节：

```
[ COBS 编码后的帧主体 ]
[ 0x00 分隔符 ]
```

**特性**：
- 最差情况下的开销：`ceil(n/254)` 字节。
- 保证编码块内部不存在 `0x00` 字节。
- 单次处理即可完成编解码，无需查找表。
- 末尾的 `0x00` 分隔符保证了在丢帧后能重新同步。
- **盲覆盖兼容性 (Blind Overwrite Compatibility)**：即使帧在环形缓冲区中被部分覆盖，通过 `0x00` 分隔符仍能可靠地定位下一帧的起始位置。

**注意**：COBS 应用于 **整个帧主体**（报头 + 时间戳 + 有效载荷长度 + 有效载荷 + CRC）。分隔符 **不属于** COBS 块。

---

## 4. CRC-16

- **算法**：CRC-16/CCITT-FALSE (`poly = 0x1021`, `init = 0xFFFF`, 无反射, 最终 XOR `0x0000`)。
- **覆盖范围**：报头 (8B) + 时间戳 (变长) + 有效载荷长度 (1B) + 有效载荷 (N 字节)。
- **计算**：预置 256 字节 ROM 查找表，实现 O(n) 处理速度。
- **校验**：解码器在相同范围内重新计算 CRC，并与末尾的 2 字节进行比较。不匹配则标记为 `FRAME_INVALID`。
- **错误恢复**：在“盲覆盖”场景下，CRC 校验失败允许主机检测到被部分覆盖的损坏帧，并安全地将其丢弃，而不会导致后续帧的同步丢失。

---

## 5. 传输差异

| 传输方式    | COBS | CRC  | 分隔符    | 备注                                     |
|-------------|------|------|-----------|------------------------------------------|
| UART        | 是   | 是   | `0x00`    | 每个 DMA 描述符或 IRQ 对应一个完整帧     |
| USB CDC     | 是   | 是   | `0x00`    | Bulk IN 端点，64 字节块传输              |
| 内存转储    | 否   | 是   | 无        | 顺序写入 RAM/Flash 区域                  |
| SWO/ITM     | 否   | 否   | 无        | 32 位刺激词流 (Stimulus Words)           |
| SEGGER RTT  | 否   | 可选 | 无        | 上行通道二进制块                         |
| CAN-FD      | 否   | 是   | 无        | 跨多个 ID 分帧；执行大端序交换           |

**规则**：
- 后端不得定义私有协议，仅能应用传输层特定的封装。
- 帧主体（报头 + 时间戳 + 有效载荷长度 + 有效载荷 + CRC）在所有传输方式下保持一致。
- SWO/ITM 省略 CRC，因为假设调试探针通道内的传输是无损的。

## 6. 解码器行为

### 6.1 有效帧

当且仅当满足以下条件时，帧才有效：
1. `sync == 0xA5`
2. `version` 的高 4 位（主版本号）受支持（当前为 `0x1`）
3. `level` 的高 4 位为 `0`
4. 时间戳可解码（报头后的第一个字节标示长度 1/2/3/5）
5. 在读取 CRC 前，实际读取的有效载荷字节数与 `payload_len` 相符
6. 在报头 + 时间戳 + 有效载荷长度 + 有效载荷上重新计算的 CRC-16 与末尾 2 字节一致

### 6.2 无效帧处理

在任何校验失败时，解码器必须：
1. 显式拒绝该帧（返回 `FRAME_INVALID` 或等效值）。
2. 跳进到下一个 `0x00` 分隔符（COBS 传输）或下一个 `0xA5` 同步字节。
3. 从下一个候选帧恢复解码。
4. **绝不崩溃或进入未定义状态**。

### 6.3 未知类型标签

如果解码器在有效载荷中遇到无法识别的类型标签（范围 `0x0A-0x7F`）：
1. 根据该类型标签的已知长度（如果已知）跳过该字段。
2. 如果长度未知，则根据启发式方法跳至下一个字段边界，或将有效载荷标记为 `PARTIAL_DECODE`。
3. 不要崩溃。

---

## 7. 版本控制

报头中的 `version` 字节编码为 `major << 4 | minor`。

- 解码器 **必须拒绝** 不支持的 **主版本号 (Major)**。
- **次版本号 (Minor)** 的增加是可以安全忽略的（新增类型标签或新增保留字段）。

当前版本：**0x10** (v1.0)。
